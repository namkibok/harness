# Agent Team Design Patterns

## 실행 모드: Cursor에서의 협업 (멀티 Task vs 순차 Task)

Claude Code의 `TeamCreate` / `SendMessage` / 공유 작업 목록 API는 **Cursor에 없다.** 동일한 설계 의도는 **`Task` 도구 + 메인 오케스트레이션 + `_workspace/` 파일 프로토콜**로 구현한다. 요약 매핑은 `cursor-runtime.md` 참조.

### 패턴 A — 멀티 Task 병렬 (팬아웃/실시간 협업에 가장 가까움)

메인 에이전트가 **동일 턴에서 `Task`를 여러 번** 호출하고 `run_in_background: true`로 병렬 실행한다. 서브에이전트는 **서로 메시지를 보낼 수 없으므로**, “발견 공유”는 `_workspace/`의 약속된 파일(예: `findings_{role}.md`)에 기록하고, 통합 Phase에서 메인이 Read·병합한다.

```
[메인 오케스트레이터]
    ├── Task(role-a) → _workspace/01_a.md
    ├── Task(role-b) → _workspace/01_b.md   (병렬)
    └── Read/통합 → 최종 산출물
```

**특징:** 파일 기반으로 교차 검증·발견 공유를 **근사**한다. 메인이 라운드마다 브리핑 파일을 써 주면 품질이 오른다.

**제약:** 동시 Task 수·컨텍스트 한도 — 통상 **2~5개** 병렬을 넘기지 않는 것이 안전하다.

**Phase 재구성:** 전문가 조합이 바뀌면 이전 `_workspace/`를 보존한 뒤, 다음 Phase용 Task 세트와 파일 규칙을 새로 정의한다.

### 패턴 B — 순차 Task (경량·파이프라인)

메인이 Task를 **한 번에 하나** 호출하거나, 이전 산출물 경로를 다음 Task 프롬프트에 넣어 순서를 강제한다.

```
[메인] → Task(분석) → artifact1 → Task(구현) → artifact2
```

**특징:** 조율이 단순하고 디버깅이 쉽다.

**제약:** 병렬 이점이 적다.

### 모드 선택 의사결정 트리 (Cursor)

```
독립적인 관점/병렬 작업으로 나눌 수 있는가?
├── Yes → 패턴 A (멀티 Task) + _workspace/ 규약 + 메인 통합
│         (Claude의 “에이전트 팀”에 해당하는 목표)
│
└── No → 패턴 B (순차 Task) 또는 단일 Task
```

> **핵심 원칙:** 원본 하네스의 “에이전트 팀 우선”은 Cursor에서 **「멀티 Task + 파일 프로토콜 + 메인 통합」 우선**으로 해석한다. 진짜 peer-to-peer 채팅은 불가에 가깝다.

## 역할 협업 아키텍처 유형 (Cursor에서는 멀티 Task + 파일로 구현)

### 1. 파이프라인 (Pipeline)
순차적 작업 흐름. 이전 에이전트의 출력이 다음 에이전트의 입력.

```
[분석] → [설계] → [구현] → [검증]
```

**적합한 경우:** 각 단계가 이전 단계의 산출물에 강하게 의존
**예시:** 소설 집필 — 세계관 → 캐릭터 → 플롯 → 집필 → 편집
**주의:** 병목이 전체 파이프라인을 지연시킴. 각 단계를 가능한 독립적으로 설계할 것.
**Cursor 구현:** 순차 Task가 기본. 파이프라인 내 병렬 구간만 멀티 Task로 당긴다.

### 2. 팬아웃/팬인 (Fan-out/Fan-in)
병렬 처리 후 결과 통합. 독립적 작업을 동시 수행.

```
         ┌→ [전문가A] ─┐
[분배] → ├→ [전문가B] ─┼→ [통합]
         └→ [전문가C] ─┘
```

**적합한 경우:** 동일 입력에 대해 서로 다른 관점/영역의 분석이 필요
**예시:** 종합 리서치 — 공식/미디어/커뮤니티/배경 동시 조사 → 통합 보고
**주의:** 통합 단계의 품질이 전체 품질을 결정.
**Cursor 구현:** **멀티 Task 병렬 + `_workspace/`** 가 가장 자연스럽다. 발견 공유는 공유 노트 파일 또는 메인이 중간에 작성하는 브리핑으로 대체한다. 단독 순차 Task보다 교차 정보가 잘 섞인다.

### 3. 전문가 풀 (Expert Pool)
상황에 따라 적절한 전문가를 선택 호출.

```
[라우터] → { 전문가A | 전문가B | 전문가C }
```

**적합한 경우:** 입력 유형에 따라 다른 처리가 필요
**예시:** 코드 리뷰 — 보안/성능/아키텍처 전문가 중 해당 영역만 호출
**주의:** 라우터의 분류 정확도가 핵심.
**Cursor 구현:** 필요한 역할만 골라 **단발 Task**로 호출하는 것이 비용 대비 유리하다.

### 4. 생성-검증 (Producer-Reviewer)
생성 에이전트와 검증 에이전트가 쌍으로 동작.

```
[생성] → [검증] → (문제시) → [생성] 재실행
```

**적합한 경우:** 산출물의 품질 보장이 중요하고 객관적 검증 기준이 존재
**예시:** 웹툰 — artist 생성 → reviewer 검수 → 문제 패널 재생성
**주의:** 무한 루프 방지를 위해 최대 재시도 횟수(2~3회) 설정 필수.
**Cursor 구현:** 순차 Task(생성 → 검증) 또는 한 라운드 검증 결과를 `_workspace/`에 기록 후 생성 Task 재호출.

### 5. 감독자 (Supervisor)
중앙 에이전트가 작업 상태를 관리하며 하위 에이전트에 동적으로 작업을 분배.

```
         ┌→ [워커A]
[감독자] ─┼→ [워커B]    ← 감독자가 상태를 보고 동적 분배
         └→ [워커C]
```

**적합한 경우:** 작업량이 가변적이거나 런타임에 작업 분배를 결정해야 할 때
**예시:** 대규모 코드 마이그레이션 — 감독자가 파일 목록을 분석하고 워커들에게 배치 할당
**팬아웃과의 차이:** 팬아웃은 사전에 작업을 고정 분배, 감독자는 진행 상황을 보며 동적 조정
**주의:** 감독자가 병목이 되지 않도록 위임 단위를 충분히 크게 설정.
**Cursor 구현:** 메인이 작업 큐를 유지하고 `Task`를 반복 호출해 워커에 할당. 진행률은 `_workspace/` 상태 파일로 추적.

### 6. 계층적 위임 (Hierarchical Delegation)
상위 에이전트가 하위 에이전트에 재귀적으로 위임. 복잡한 문제를 단계적으로 분해.

```
[총괄] → [팀장A] → [실무자A1]
                  → [실무자A2]
       → [팀장B] → [실무자B1]
```

**적합한 경우:** 문제가 자연스럽게 계층적으로 분해되는 구조
**예시:** 풀스택 앱 개발 — 총괄 → 프론트엔드팀장 → (UI/로직/테스트) + 백엔드팀장 → (API/DB/테스트)
**주의:** 깊이 3단계 이상은 지연과 컨텍스트 손실이 커짐. 2단계 이내 권장.
**Cursor 구현:** Task 안에서 또 Task를 호출하는 중첩은 피하고, **평탄화된 역할 목록 + 메인이 단계별로 Task를 순차/병렬 호출**하는 편이 안전하다.

## 복합 패턴

실전에서는 단일 패턴보다 복합 패턴이 흔하다:

| 복합 패턴 | 구성 | 예시 |
|----------|------|------|
| **팬아웃 + 생성-검증** | 병렬 생성 후 각각 검증 | 다국어 번역 — 4개 언어 병렬 번역 → 각각 네이티브 리뷰어 검수 |
| **파이프라인 + 팬아웃** | 순차 단계 중 일부를 병렬화 | 분석(순차) → 구현(병렬) → 통합 테스트(순차) |
| **감독자 + 전문가 풀** | 감독자가 전문가를 동적 호출 | 고객 문의 처리 — 감독자가 문의 분류 후 적합한 전문가 할당 |

### 복합 패턴에서의 실행 모드

**기본적으로 복합 패턴에는 멀티 Task + `_workspace/`를 우선한다.** Cursor에서는 “커뮤니케이션”이 곧 **파일·메인 브리핑**이다.

| 시나리오 | 권장 모드 | 이유 |
|---------|----------|------|
| **리서치 + 분석** | 멀티 Task 병렬 | 관점별 산출물을 파일로 모은 뒤 메인이 상충 정리 |
| **설계 + 구현 + 검증** | 순차 + 필요 시 병렬 | 단계 의존이 강하면 순차, 독립 모듈은 병렬 |
| **감독자 + 워커** | 메인 루프 + 반복 Task | 메인이 큐·상태 파일을 보고 다음 Task를 스폰 |
| **생성 + 검증** | 순차 Task 또는 짧은 재시도 루프 | 검증 산출물을 `_workspace/`에 남기고 생성 Task에 재주입 |

> 단일 Task만으로 충분한 **완전 단발·격리 작업**은 경량 순차 1회로 둔다.

## 에이전트 타입 선택

`Task` 호출 시 `subagent_type`(및 기타 지원 파라미터)으로 역할 성격을 맞춘다. Cursor 빌트인 타입 이름은 환경에 따라 다를 수 있으므로, 실제 호출 가능한 값은 에디터/문서를 따른다.

### 자주 쓰는 Task 타입 (개념)

| 타입(예시) | 특성 | 적합한 용도 |
|------|----------|-----------|
| `generalPurpose` | 범용 도구 사용 가능에 가깝게 동작 | 구현, 웹 조사, 복합 작업 |
| `explore` | 탐색·분석에 치우침 (편집 최소화) | 코드베이스 읽기·리뷰 |
| `shell` | 터미널 중심 | 빌드, 테스트, 스크립트 실행 |

### 역할 명세 파일 (커스텀)

`.cursor/harness/agents/{name}.md`에 역할·원칙·입출력을 적어 두고, Task 프롬프트에서 **해당 파일을 Read하도록 지시**하거나 요약을 인라인으로 넣는다. Cursor는 파일 이름이 곧 `subagent_type`으로 자동 등록되지 않을 수 있다.

### 선택 기준

| 상황 | 권장 | 이유 |
|------|------|------|
| 역할이 복잡하고 재사용 | **`.cursor/harness/agents/*.md` + Task** | 페르소나와 프로토콜을 버전 관리 |
| 단순 일회성 | **`generalPurpose` + 짧은 프롬프트** | 파일 없이도 가능 |
| 읽기 위주 분석/리뷰 | **`explore`** | 불필요한 편집 위험 감소 |
| 계획·설계만 | **`generalPurpose` 또는 `explore` + “코드 변경 금지” 프롬프트** | Cursor 전용 `Plan` 타입에 의존하지 않음 |
| 구현·리팩터 | **`generalPurpose`** | 편집·실행 필요 |

**원칙:** 하네스로 만든 **전문 역할은 모두 `.cursor/harness/agents/{name}.md`에 남긴다.** 다음 세션·다른 에이전트가 동일 기대치를 공유하게 한다.

**모델:** Task에 `model`을 줄 수 있으면 **난이도 높은 위임에 유리한 모델**을 지정한다.

## 에이전트 정의 구조

```markdown
---
name: agent-name
description: "1-2문장 역할 설명. 트리거 키워드 나열."
---

# Agent Name — 역할 한줄 요약

당신은 [도메인]의 [역할] 전문가입니다.

## 핵심 역할
1. 역할1
2. 역할2

## 작업 원칙
- 원칙1
- 원칙2

## 입력/출력 프로토콜
- 입력: [어디서 무엇을 받는지]
- 출력: [어디에 무엇을 쓰는지]
- 형식: [파일 포맷, 구조]

## 협업 프로토콜 (Cursor · 멀티 Task 모드)
- **파일 수신:** [어떤 `_workspace/` 경로를 읽는지]
- **파일 발신:** [어떤 경로에 어떤 스키마로 쓰는지]
- **메인 중재:** [메인이 합쳐 줄 브리핑/이슈 목록 파일이 있는지]

## 에러 핸들링
- [실패 시 행동]
- [타임아웃 시 행동]

## 협업
- 다른 에이전트와의 관계
```

## 에이전트 분리 기준

| 기준 | 분리 | 통합 |
|------|------|------|
| 전문성 | 영역이 다르면 분리 | 영역이 겹치면 통합 |
| 병렬성 | 독립 실행 가능하면 분리 | 순차 종속이면 통합 고려 |
| 컨텍스트 | 컨텍스트 부담이 크면 분리 | 가볍고 빠르면 통합 |
| 재사용성 | 다른 팀에서도 쓰면 분리 | 이 팀에서만 쓰면 통합 고려 |

## 스킬 vs 에이전트 구분

| 구분 | 스킬 (Skill) | 에이전트 (Agent) |
|------|-------------|-----------------|
| 정의 | 절차적 지식 + 도구 번들 | 전문가 페르소나 + 행동 원칙 |
| 위치 | `.cursor/skills/` | `.cursor/harness/agents/` |
| 트리거 | 사용자 요청과 description 매칭 | 메인이 Task로 명시적 위임 |
| 크기 | 작은~큰 (워크플로우) | 작은 (역할 정의) |
| 용도 | "어떻게 하는가" | "누가 하는가" |

스킬은 에이전트가 작업을 수행할 때 참조하는 **절차적 가이드**.
에이전트는 스킬을 활용하는 **전문가 역할 정의**.

## 스킬 ↔ 에이전트 연결 방식

에이전트가 스킬을 활용하는 3가지 방식:

| 방식 | 구현 | 적합한 경우 |
|------|------|-----------|
| **스킬 트리거** | Cursor가 해당 스킬을 로드한 뒤 지침 준수 | 사용자 요청이 스킬 description과 맞을 때 |
| **프롬프트 내 인라인** | 역할 `.md`에 스킬 핵심을 짧게 포함 | 50줄 이하·역할 전용일 때 |
| **레퍼런스 로드** | `Read`로 `.cursor/skills/.../references/` 를 필요 시 로드 | 대용량·조건부일 때 |

권장: 재사용성이 높으면 프로젝트 스킬로 분리, 전용이면 인라인, 대용량이면 references/ 로드.
